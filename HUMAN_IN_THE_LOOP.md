## 目的

このドキュメントは、エージェントが独断で進めないための承認ポイントを定義します。
該当する場合は作業を止めて、人間の明示的な判断を待ってください。

## 事前承認が必要な操作

- 公開 API の削除・引数変更・戻り値仕様変更などの後方互換性を壊す変更
- Ruby の最低バージョン変更
- 新規 runtime 依存の追加
- 破壊的操作（例: `git reset --hard`, 大量削除）
- CI やリリースフローの仕様変更
- 大規模なリファクタリング（機能追加やバグ修正に直接関係しない広範囲変更）

## 相談してから進めるべきケース

- 振る舞い変更を伴うが、期待仕様がテストや README から一意に読めない場合
- 複数の実装方針があり、トレードオフが明確に存在する場合
- 既存の記述（README / spec / code コメント）間で矛盾がある場合

## 承認の取り方

- 何を変えるか（変更点）
- なぜ必要か（背景と問題）
- 影響範囲（互換性、利用者影響、CI・ドキュメント）
- 代替案（あれば）

上記を短く提示し、承認後に実装へ進みます。
